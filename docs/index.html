トラブルシューティング

cmake_minimum_required の互換性エラー（CMake 4.x）

CMake 4.x と古い依存ライブラリの cmake_minimum_required 宣言が衝突して起きます。cmake/Dependencies.cmake 内で CMAKE_POLICY_VERSION_MINIMUM 3.5 を設定済みのため本プロジェクトでは回避済みです。

macOS でカメラが開かない（Chapter 1 以降）

システム設定 → プライバシーとセキュリティ → カメラ で、使用しているターミナルアプリ（Terminal.app / iTerm2 / VS Code）に許可を与えます。初回起動時にダイアログが出ない場合は手動で設定が必要です。

ビルドが壊れた感じがする（キャッシュ破損）

CMake のキャッシュが破損したら rm -rf build して最初から configure し直します。TypeScript の rm -rf node_modules && npm install に相当します。

深掘り 00a: なぜヘッダが必要なのか
TL;DR
  • C++ のコンパイラは 1ファイル（= Translation Unit） だけを見る。他のファイルの中身は見えない。
  • 関数を呼ぶには シグネチャ（戻り値・引数） を知っている必要がある。
  • この「シグネチャだけ先に教えるファイル」が ヘッダ（.hpp / .h）。
  • tsc は全ファイルを舐めるが、C++ は 1 ファイルずつ独立でコンパイル → リンカが結合。

コンパイル・リンクのモデル図

greeting.cpp ──┐
               ├── clang++ ──► greeting.o ──┐
greeting.hpp ──┘                             ├── ld ──► chapter00_ref_app
                                             │
main.cpp ────┐                               │
             ├── clang++ ──► main.o ─────────┘
greeting.hpp ─┘

greeting.hpp は両方のコンパイル単位に貼り込まれます。両 .o に「この関数が存在する」という前提が埋め込まれ、リンク時に本体と結合されます。

#include は「コピペ」

プリプロセッサが .hpp のテキストをその場に貼り付ける命令です。ヘッダは軽く保つべきの理由はここにあります。重いヘッダを include すると、インクルードした全ファイルがその分重くコンパイルされます。

#pragma once が守っていること

C++ 
// a.hpp (pragma once なし)
struct Foo { int x; };

// main.cpp
#include "a.hpp"  // Foo が展開される
#include "b.hpp"  // b.hpp が a.hpp を include → Foo が2回定義 → エラー

ODR — One Definition Rule

非 inline な関数・変数の定義は、プログラム全体でちょうど1回でなければならない。宣言は何回あってもよい。
ヘッダに書いていいもの理由
宣言（関数シグネチャ）定義ではないため何度でも可
inline / constexpr 関数各 TU に定義があっても OK と規格が保証
テンプレート実体化のため定義が見える必要がある
クラス定義宣言扱い（メンバ関数の実装は .cpp に）
inline 変数（C++17〜）同上

再コンパイルコストの観点

.hpp を 1 行変えると、それを #include した全ての .cpp が再コンパイルされます。.cpp を 1 行変えても、その .cpp だけ再コンパイルすれば OK。大規模プロジェクトでは「ヘッダは最小・実装詳細は .cpp に押し込む」が鉄則です。

Tip: C++20 Modules への展望 C++20 では import std; のような Modules が導入され、ヘッダのコピペ問題を根本から解決します。ただしコンパイラ・ビルドシステム双方のサポートがまだ発展途上のため、本プロジェクトでは伝統的なヘッダ方式を採用しています。
深掘り 00b: CMake の2段階実行メンタルモデル
TL;DR
  • CMake は Configure → Generate → Build の 3 段階で動く
  • CMakeLists.txt はスクリプト（宣言的ではない）。上から順に実行される

この章で学ぶこと

C++ 概念説明
RAIIコンストラクタでリソース取得、デストラクタで解放。finally 不要の理由
= delete物理デバイスを2つのオブジェクトが所有する問題を型で防ぐ
= default（ムーブ）所有権の移転。コピーせず「持ち主を変える」

find_package(OpenCV)システムにインストール済みのライブラリを CMake で参照

なぜ RAII ラッパが必要か

C++ 
// NG: 例外が飛んだら release() が呼ばれない
cv::VideoCapture cap;
cap.open(0);
// ... 何か処理 ...
cap.release(); // ← 例外で到達しないことがある

// OK: RAII ラッパで自動解放
{
    media::CameraCapture cam;
    cam.open(0);
    // ... 何か処理 ...
} // ← ここで ~CameraCapture() が自動実行

深掘り 01: RAII とデストラクタ
TL;DR
  • C++ ではオブジェクトがスコープを出ると必ずデストラクタが呼ばれる（例外が飛んでも保証）
  • 「コンストラクタでリソース取得、デストラクタで解放」= RAII
  • TS の finally や Go の defer と似た目的を、言語のオブジェクトモデルで実現
  • コピー禁止（= delete）+ ムーブのみ許可（= default）で所有権を型で表現できる

5つの特殊メンバ関数

関数役割
デフォルトコンストラクタ T()オブジェクト初期化
コピーコンストラクタ T(const T&)別オブジェクトからコピー
コピー代入 T& operator=(const T&)代入でコピー
ムーブコンストラクタ T(T&&)別オブジェクトから所有権移転
ムーブ代入 T& operator=(T&&)代入で所有権移転

Rule of Five と Rule of Zero

C++ 
class CameraCapture {
    // Rule of Five: デストラクタを自前で書いたら残り4つも定義する
    ~CameraCapture();                              // 自前で書く
    CameraCapture(const CameraCapture&) = delete; // コピー禁止を明示
    CameraCapture& operator=(const CameraCapture&) = delete;
    CameraCapture(CameraCapture&&) = default;     // ムーブはOK
    CameraCapture& operator=(CameraCapture&&) = default;
};

コピー禁止にする理由

カメラ（物理デバイスハンドル）を2つのオブジェクトが所有すると、両方のデストラクタが cap_.release() を呼び二重解放になります。= delete でコンパイルエラーにして型レベルで防ぎます。

RAII の恩恵

問題finally / deferRAII
リソースリーク防止○（手動）○（自動）
例外安全○（try-finally）○（stack unwinding）
早期 return での安全△（書き忘れリスク）○（自動）
所有権の型表現△（慣習依存）○（= delete / unique_ptr）
Tip: C++23 の std::unique_resource カスタムリソース（POSIX ファイルディスクリプタなど）を RAII 化する際、C++23 では std::unique_resource を使うと自前でクラスを書かずに済みます。

Ch.2 HSV色域トラッカー

特定色の物体の重心と面積を検出して画面に描画します。std::span・std::optional・std::ranges と、値渡し/参照渡し/ムーブの使い分けを学びます。

この章で学ぶこと

C++ 概念説明
std::optional<T>「検出できた/できなかった」を型で表現。null チェックより安全
std::span<const T>非所有ビュー。大きなデータをコピーせず「借りる」
std::rangesパイプライン的な配列処理。TS の Array メソッドチェーンに相当
const T& vs T参照渡しとコピーの使い分け。cv::Mat は参照カウントを持つ

TrackResult と HsvRange

C++ 
struct HsvRange {
    cv::Scalar lower; // HSV 下限値（H: 0-179, S: 0-255, V: 0-255）
    cv::Scalar upper;
};

struct TrackResult {
    cv::Point2f centroid; // 重心座標 (pixel, float 精度)
    double area;          // 検出領域の面積 (pixel^2)
};

// track 関数: 検出失敗時は nullopt を返す
[[nodiscard]] std::optional<TrackResult> track(
    const cv::Mat& bgr,    // const& で「借りる」。コピーしない
    const HsvRange& range
);

cv::Mat の参照カウントと const& の関係

cv::Mat は内部で参照カウントを持ち、= 代入はシャローコピー（バッファ共有）です。const cv::Mat& で渡すと参照カウントも増えないため最も効率的です。独立コピーが必要な場合は .clone() を明示的に呼びます。

Tip: OpenCV の HSV 色空間の罠 OpenCV の H（色相）は 0〜179（実際の 0〜360 度を半分にスケール）です。赤は 0〜10 と 170〜179 の両端にまたがるため、赤の検出には2回 inRange を呼んで結果を OR する必要があります。

深掘り 02: ムーブセマンティクス
TL;DR
  • コピー: 値を複製する。元のオブジェクトはそのまま。コストが高い場合がある
  • ムーブ: 所有権（内部リソース）を移転する。元は「空っぽ」になる。O(1) が多い
  • 参照（const T&）: 所有権も複製もしない。借りるだけ
  • cv::Mat は参照カウント付き。コピーしても実際はポインタのコピーのみ（浅いコピー）

TS との根本的な違い

TypeScript vs C++ 
// TS: オブジェクトの代入は参照コピー
const a = { x: 1 };
const b = a;    // b と a は同じオブジェクトを指す
b.x = 2;        // a.x も 2 になる！

// C++: 代入はデフォルトで値コピー
struct Point { int x; };
Point a{ 1 };
Point b = a;    // b は a とは独立したコピー

C++ 
// OSC 1.0 がサポートする型を variant で列挙
using OscValue = std::variant<int, float, std::string, bool>;

// if constexpr で型ごとの処理を書く（コンパイル時分岐）
std::string format(const OscValue& v) {
    return std::visit([](const auto& val) -> std::string {

        else return "?";
    }, v);
}

TS discriminated union との対比

// TS: discriminated union
type OscValue =
    | { kind: 'int';    value: number }
    | { kind: 'float';  value: number }
    | { kind: 'string'; value: string };

        if constexpr (std::is_same_v<T, float>)       return "f:" + std::to_string(v);

C++

Detector concept の定義

C++ 
template <typename T>
concept Detector = requires(T t, const cv::Mat& frame) {

深掘り 04: concepts — TS の interface と C++ concept の違い
TL;DR
  • C++ の concept は「型がある操作をサポートするか」をコンパイル時に検査する
  • TS の interface と似ているが、継承不要・vtable なし・実行時コストゼロ
  • 仮想関数は「動的ディスパッチ」（実行時に決まる）
  • concept + テンプレートは「静的ディスパッチ」（コンパイル時に決まる）

TS interface との比較

TypeScript → C++ 
// TS: interface + implements（継承）
interface Detector {
    process(frame: Mat): DetectorOutput | null;
}
class ColorTracker implements Detector { ... }

// C++: concept（継承なし。process() を定義するだけで OK）
template <typename T>
concept Detector = requires(T t, const cv::Mat& frame) {
    { t.process(frame) } -> std::same_as<std::optional<DetectorOutput>>;
};

template <Detector T>
void run_detector(T& detector, const cv::Mat& frame) {
    auto result = detector.process(frame); // コンパイル時に型が確定
}

仮想関数concept + template
ディスパッチ動的（実行時）静的（コンパイル時）
オーバーヘッドvtable ルックアップ（〜数ns）ゼロ（インライン化可能）

コードサイズバイナリが小さいテンプレート実体化で増える

前フレーム保持の設計

C++ 
class OpticalFlowDetector {
    // optional: 初回フレームがまだない状態を nullopt で表現
    std::optional<cv::Mat> prev_gray_;

public:
    std::optional<DetectorOutput> process(const cv::Mat& frame) {
        cv::Mat gray;
        cv::cvtColor(frame, gray, cv::COLOR_BGR2GRAY);

        if (!prev_gray_) {
            prev_gray_ = gray.clone(); // ← clone() 必須。浅いコピーは NG
            return std::nullopt; // 初回は結果なし
        }
        // フロー計算 → DetectorOutput を返す...
    }
};
Tip: Dense vs Sparse オプティカルフロー Farneback 法（Dense）は全ピクセルのフローを計算するため情報量が多いですが重いです。Lucas-Kanade 法（Sparse）は特徴点のみ追うため軽量です。本プロジェクトでは「動き量の平均」を OSC で送るだけなので Dense で十分ですが、60fps を要求する場合は Sparse への切り替えを検討してください。

典型的な配置

C++

深掘り 08: std::span と C API の相互運用
TL;DR
  • std::span<T> は「メモリの所有権を持たない、連続したデータへのビュー」
  • TS の Uint8Array / ArrayBufferView に相当する
  • ONNX Runtime などの C API を安全に扱うブリッジとして活用
  • PIMPL パターンで外部ライブラリへの依存をヘッダから隠せる

よくあるビルドエラー Q&A

error: use of undeclared identifier 'xxx'

ヘッダの #include 忘れ、または名前空間の指定忘れが多いです。使いたい型/関数のヘッダを確認してください。

undefined reference to 'xxx'

コンパイルは通るがリンクで失敗しています。target_link_libraries でライブラリのリンクが抜けているか、実装 (.cpp) がビルドに含まれていない可能性があります。

error: call to deleted function

= delete されたコピーコンストラクタ/代入演算子を呼ぼうとしています。std::move() でムーブするか、設計を見直してください。

error: no matching function for call to ...

オーバーロード解決に失敗しています。引数の型、const 修飾子、テンプレート制約（concept）を確認してください。

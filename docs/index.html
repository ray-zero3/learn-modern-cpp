学習フィロソフィー — なぜこのアプローチか

TL;DR
  • 動かしながら学ぶ: 各章は「画面に何かが映る」「データがネットワークに飛ぶ」などの具体的な成果物で終わる
  • TypeScript アナロジーで橋渡し: 抽象的な概念を「TSでいうと○○」で噛み砕いてから深掘りする
  • C++ 固有の罠を正面から扱う: 所有権・未定義動作・リンカなど、TSでは見えない層を意図的に解説する

TypeScript 開発者が C++ でつまずく典型的なポイントは、構文ではなく概念の断絶にあります。

TypeScript の前提C++ で崩れる前提
変数への代入は「参照のコピー」値型（int, struct）への代入は値のコピー。別オブジェクトになる
GC がメモリを管理するスタック・ヒープ・所有権を開発者が明示的に選ぶ
実行時エラーがほとんど例外で捕捉できる配列外アクセス・ダングリング参照などは未定義動作でサイレントに壊れる
import は解決済み（全ファイルが見える）コンパイルは1ファイルずつ独立。リンカが後で結合する
型エラーはコンパイル時に確定テンプレートエラーは実体化時に発覚。エラーメッセージが長大になりやすい
関数のコールコストはほぼ均一仮想関数コール（vtable 間接参照）と直接呼び出しには実測可能な差がある

この教材の読み方

  1. まず「環境構築」を完了させる — cmake --build build が通るまで先に進まない。ビルドが壊れた状態で学んでも無意味。
  2. 各章は「達成チェックリスト」から読む — ゴールを先に把握してから本文を読むと効率的。
  3. コードは必ず手で写す — reference/ のコードをそのまま使わず、workspace/ に自分で実装してみる。
  4. 深掘りは「引っかかったとき」に開く — 最初から全部読む必要はない。「なぜそうなのか」が気になったタイミングで開く。
  5. エラーメッセージを恐れない — C++ のエラーは長いが、最初の数行が本質。clang++ は GCC より親切なメッセージを出す。
  6. 章をまたいで進めない — 各章は前章の成果物を前提とする。Chapter 1 が完全に動いてから Chapter 2 へ。
TypeScript 開発者が C++ を学ぶ最大のメリット TS では「隠蔽されている」メモリモデル・コンパイルパイプライン・リンカの仕組みを理解することで、TS の動作についてもより深く理解できます。たとえば「なぜ import が循環するとバンドラが困るのか」「なぜ型定義ファイル(.d.ts)が必要なのか」が C++ の仕組みを通じて自然に分かるようになります。

cmake フラグの詳細説明

CMake のフラグは TS の tsconfig.json の各フィールドに相当します。

フラグ意味TS 相当
-S .ソースディレクトリ（CMakeLists.txt がある場所）プロジェクトルート
-B buildビルド成果物を置くディレクトリ（自動作成される）dist/ や out/
-DCMAKE_CXX_COMPILER=...使用するコンパイラのパスを指定package.json の engines フィールドに相当
-DCMAKE_BUILD_TYPE=Debugビルド構成（Debug/Release など）NODE_ENV=development
-GNinjaビルドシステムのジェネレータを Ninja に指定npm vs yarn の選択に近い
-DCMAKE_EXPORT_COMPILE_COMMANDS=ONcompile_commands.json を出力（clangd が必要）tsconfig の paths 等の IDE 連携情報

Out-of-source build とは

CMake ではソースディレクトリとビルドディレクトリを分けるのが標準的な作法です。これを "out-of-source build" と呼びます。

In-source build（非推奨）Out-of-source build（推奨）
ビルド成果物ソースと同じ場所に混在するbuild/ ディレクトリに集約される
クリーン個別に削除が必要rm -rf build 一発でリセット
.gitignore複数パターンを書く必要があるbuild/ を1行追加するだけ
複数構成の共存不可（衝突する）build-debug/ build-release/ を並列で持てる
Tip: Ninja でビルドを高速化 brew install ninja した後、cmake に -GNinja を追加すると並列ビルドが効きやすく数倍速くなります。デフォルトの Makefile ジェネレータより依存解析が賢く、差分ビルドが高速です。

ビルドタイプ

CMAKE_BUILD_TYPEコンパイルフラグ用途

compile_commands.json とは

compile_commands.json は「各ソースファイルをどのコンパイラコマンドでビルドするか」を記録した JSON ファイルです。clangd（言語サーバ）はこれを読み込むことで、実際のビルドと同じ include パス・マクロ定義・言語規格で型チェック・補完を提供します。

json（compile_commands.json の例） 
[
  {
    "directory": "/path/to/modern-cpp/build",
    "command": "/opt/homebrew/opt/llvm/bin/clang++ -std=c++20 -I../include ... -c ../src/greeting.cpp",
    "file": "../src/greeting.cpp"
  }
]

VS Code では clangd 拡張（llvm-vs-code-extensions.vscode-clangd）をインストールし、clangd.path に /opt/homebrew/opt/llvm/bin/clangd を指定します。TypeScript の tsserver に相当する役割を果たします。

Tip: AddressSanitizer でメモリバグを早期発見 
cmake -S . -B build-asan \
  -DCMAKE_CXX_COMPILER=/opt/homebrew/opt/llvm/bin/clang++ \
  -DCMAKE_BUILD_TYPE=Debug \
  -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined"
cmake --build build-asan
-fsanitize=address は境界外アクセス・use-after-free を、undefined は未定義動作を実行時に検出します。開発中は常に有効にすることを推奨します。TypeScript に相当するものがないほど強力なデバッグツールです。
Tip: ccache でリビルドを高速化 brew install ccache → cmake に -DCMAKE_CXX_COMPILER_LAUNCHER=ccache を追加するだけ。2回目以降のビルドがキャッシュヒットして大幅に速くなります。ファイルを変更していない Translation Unit は再コンパイルをスキップします。

トラブルシューティング

cmake_minimum_required の互換性エラー（CMake 4.x）

CMake 4.x と古い依存ライブラリの cmake_minimum_required 宣言が衝突して起きます。cmake/Dependencies.cmake 内で CMAKE_POLICY_VERSION_MINIMUM 3.5 を設定済みのため本プロジェクトでは回避済みです。

macOS でカメラが開かない（Chapter 1 以降）

システム設定 → プライバシーとセキュリティ → カメラ で、使用しているターミナルアプリ（Terminal.app / iTerm2 / VS Code）に許可を与えます。初回起動時にダイアログが出ない場合は手動で設定が必要です。

確認コマンド: system_profiler SPCameraDataType でカメラデバイスが認識されているか確認できます。

ビルドが壊れた感じがする（キャッシュ破損）

CMake のキャッシュが破損したら rm -rf build して最初から configure し直します。TypeScript の rm -rf node_modules && npm install に相当します。CMakeCache.txt を手動編集するのは避けてください。

OpenCV が見つからない（could not find OpenCV）

brew install opencv でインストールし、brew --prefix opencv でパスを確認します。CMake に -DOpenCV_DIR=$(brew --prefix opencv)/lib/cmake/opencv4 を追加すると明示的にパスを指定できます。

FetchContent でダウンロードが失敗する

ネットワーク環境（プロキシ等）が原因のことが多いです。FETCHCONTENT_UPDATES_DISCONNECTED ON を CMake キャッシュに設定すると、一度ダウンロードした後はネットワーク不要で動作します。また build/_deps/ 以下にダウンロードされたソースがキャッシュされます。

CMake 3ターゲット構成

Chapter 0 の reference/CMakeLists.txt は以下の3ターゲットを作成します：

  1. chapter00_ref_greet — add_library。greeting.cpp をコンパイルして .a にする静的ライブラリ
  2. chapter00_ref_app — add_executable。greet ライブラリをリンクした実行ファイル
  3. chapter00_ref_tests — doctest をリンクしたテストバイナリ

CMakeLists.txt を行ごとに読む

CMakeLists.txt（アノテーション付き） 
# --- ライブラリターゲット ---
# add_library(名前 ソースファイル...)
# STATIC = .a（静的リンク）/ SHARED = .so(.dylib) / OBJECT = .o のみ
add_library(chapter00_ref_greet STATIC
    src/greeting.cpp   # .cpp だけ書く。.hpp は自動でヘッダ検索される
)

# include パスを公開 (PUBLIC = 自分 + リンク元に伝播)
# TS の tsconfig.json の "paths" に相当
target_include_directories(chapter00_ref_greet PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}/include
)

# C++20 を要求。target_ 系でターゲットごとに設定するのがモダン CMake の作法
target_compile_features(chapter00_ref_greet PUBLIC cxx_std_20)

# --- 実行ファイルターゲット ---
add_executable(chapter00_ref_app app/main.cpp)

# PRIVATE = 自分だけ使う（greet ライブラリの include パスも一緒に伝播する）
target_link_libraries(chapter00_ref_app PRIVATE chapter00_ref_greet)

# --- テストターゲット ---
add_executable(chapter00_ref_tests tests/test_greeting.cpp)
target_link_libraries(chapter00_ref_tests PRIVATE chapter00_ref_greet doctest::doctest)

# CTest に登録する（ctest コマンドで実行できるようになる）
add_test(NAME chapter00_ref_tests COMMAND chapter00_ref_tests)

cmake -S . -B build が内部で何をするか

このコマンドは実際には3段階のうち最初の2段階（Configure + Generate）を実行します。

  1. Configure（CMakeLists.txt の実行）: CMakeLists.txt をスクリプトとして上から順に実行し、内部モデル（ターゲット・プロパティ・依存関係のグラフ）を構築します。この段階でシステムの OpenCV・LLVM などの検索も行われます。
  2. Generate（ビルドファイルの生成）: 内部モデルを元に build.ninja（または Makefile）を生成します。CMake 自身はコンパイルしません。
  3. Build（実際のコンパイル・リンク）: cmake --build build で Ninja や Make がコンパイラを呼び出します。ここで .cpp → .o → .a / 実行ファイル の変換が起きます。

FetchContent が内部でやること

CMake 
FetchContent_Declare(
    doctest
    GIT_REPOSITORY https://github.com/doctest/doctest.git
    GIT_TAG        v2.4.11
)
FetchContent_MakeAvailable(doctest)

上記の2行が行う処理を順番に説明します：

  1. FetchContent_Declare: 「このライブラリをどこから取得するか」を登録するだけ（まだダウンロードはしない）
  2. FetchContent_MakeAvailable: 実際に git clone を実行して build/_deps/doctest-src/ に保存し、そのディレクトリをビルドに追加する（add_subdirectory と同等）
  3. 2回目以降のビルドではダウンロード済みのソースを再利用するため、ネットワーク不要

TypeScript の npm install では node_modules/ にパッケージが置かれますが、FetchContent では build/_deps/ が同等の役割を果たします。

PRIVATE / PUBLIC / INTERFACE の使い分け

指定このターゲット自身リンク元への伝搬典型用途
PRIVATE○×内部実装詳細（外部に漏らしたくない）
PUBLIC○○API の一部（呼び出し元も必要）
INTERFACE×○ヘッダオンリーライブラリ（自分はコンパイルしない）

よくあるミス: inline なしでヘッダに定義を書く（ODR 違反）

C++（ODR 違反の例） 
// greeting.hpp — NG パターン
#pragma once
#include <string>

// inline なしで関数の「定義」をヘッダに書くと、このヘッダを
// include した全 .cpp に定義が展開されてリンカが「重複定義エラー」を出す
std::string greet(const std::string& name) {  // ← NG
    return "Hello, " + name + "!";
}

// OK パターン1: inline キーワードで「複数 TU に定義があっても良い」と宣言
inline std::string greet(const std::string& name) {  // ← OK
    return "Hello, " + name + "!";
}

// OK パターン2: ヘッダには宣言のみ、定義は .cpp に書く（推奨）
std::string greet(const std::string& name); // ← 宣言のみ

「ヘッダには宣言のみ、定義は .cpp に書く」が基本ルールです。例外として、inline 関数・constexpr 関数・テンプレートはヘッダに定義を書く必要があります（定義が見えないとインライン化・実体化ができないため）。

名前空間の使い方

C++ 
// greeting.hpp
namespace greet {
    std::string greet(const std::string& name); // greet::greet()
    enum class TimeOfDay { Morning, Afternoon, Evening, Night };
}

// main.cpp — 使い方
#include "greet/greeting.hpp"  // ← TS の import に相当

int main() {
    std::cout << greet::greet("World") << "\n"; // greet:: = TS の import { greet } from "..." に相当
}

TS の import { greet } from './greeting' は実行時のモジュール解決ですが、C++ の #include はコンパイル時のテキスト展開です。名前空間はファイル境界とは独立して存在し、複数ファイルで同じ名前空間を使えます。

深掘り 00a: なぜヘッダが必要なのか
TL;DR
  • C++ のコンパイラは 1ファイル（= Translation Unit） だけを見る。他のファイルの中身は見えない。
  • 関数を呼ぶには シグネチャ（戻り値・引数） を知っている必要がある。
  • この「シグネチャだけ先に教えるファイル」が ヘッダ（.hpp / .h）。
  • tsc は全ファイルを舐めるが、C++ は 1 ファイルずつ独立でコンパイル → リンカが結合。

コンパイル・リンクのモデル図

greeting.cpp ──┐
               ├── clang++ ──► greeting.o ──┐
greeting.hpp ──┘                             ├── ld ──► chapter00_ref_app
                                             │
main.cpp ────┐                               │
             ├── clang++ ──► main.o ─────────┘
greeting.hpp ─┘

greeting.hpp は両方のコンパイル単位に貼り込まれます。両 .o に「この関数が存在する」という前提が埋め込まれ、リンク時に本体と結合されます。

#include は「コピペ」

プリプロセッサが .hpp のテキストをその場に貼り付ける命令です。ヘッダは軽く保つべきの理由はここにあります。重いヘッダを include すると、インクルードした全ファイルがその分重くコンパイルされます。

#pragma once が守っていること

C++ 
// a.hpp (pragma once なし)
struct Foo { int x; };

// main.cpp
#include "a.hpp"  // Foo が展開される
#include "b.hpp"  // b.hpp が a.hpp を include → Foo が2回定義 → エラー

ODR — One Definition Rule

非 inline な関数・変数の定義は、プログラム全体でちょうど1回でなければならない。宣言は何回あってもよい。
ヘッダに書いていいもの理由
宣言（関数シグネチャ）定義ではないため何度でも可
inline / constexpr 関数各 TU に定義があっても OK と規格が保証
テンプレート実体化のため定義が見える必要がある
クラス定義宣言扱い（メンバ関数の実装は .cpp に）
inline 変数（C++17〜）同上

再コンパイルコストの観点

.hpp を 1 行変えると、それを #include した全ての .cpp が再コンパイルされます。.cpp を 1 行変えても、その .cpp だけ再コンパイルすれば OK。大規模プロジェクトでは「ヘッダは最小・実装詳細は .cpp に押し込む」が鉄則です。

ODR 違反の典型例と診断

C++（リンカエラーが出る ODR 違反） 
// util.hpp（pragma once なし、かつ inline なし）
std::string helper() { return "hello"; } // ← 定義をヘッダに書いている

// a.cpp
#include "util.hpp" // helper() の定義が a.o に埋め込まれる

// b.cpp
#include "util.hpp" // helper() の定義が b.o にも埋め込まれる

// リンカエラー: duplicate symbol 'helper()' in a.o and b.o
// ↑ pragma once でインクルードガードしても同一 TU での二重定義は防げるが、
//   別 TU（.cpp）での重複は防げない

// 修正: inline を付ける（各 TU に定義があっても ODR OK と規格が保証）
inline std::string helper() { return "hello"; } // ← OK
Tip: C++20 Modules への展望 C++20 では import std; のような Modules が導入され、ヘッダのコピペ問題を根本から解決します。ただしコンパイラ・ビルドシステム双方のサポートがまだ発展途上のため、本プロジェクトでは伝統的なヘッダ方式を採用しています。

std::unique_ptr — ヒープオブジェクトの RAII

std::unique_ptr<T> は「ヒープに確保したオブジェクトへの唯一の所有者」を表します。unique_ptr がスコープを出るとデストラクタで delete が自動で呼ばれます。

C++ 
// NG: 手動 new/delete (例外で leakする)
cv::VideoCapture* cap = new cv::VideoCapture();
cap->open(0);
// 例外が飛ぶと delete cap; に到達しない → リーク
delete cap;

// OK: unique_ptr で自動管理
auto cap = std::make_unique<cv::VideoCapture>();
cap->open(0);
// スコープを出ると delete が自動で呼ばれる

// TS の相当物（概念的に）
// const cap = new VideoCapture();  ← GC が管理（解放タイミング不明）
// await using cap = new VideoCapture(); ← using 宣言（Explicit Resource Management）

Rule of Zero — 何も書かなくていい場合

「Rule of Zero」とは「デストラクタ・コピー/ムーブコンストラクタ・代入演算子を自分で書かなくていい設計を目指せ」という原則です。

C++ 
// Rule of Zero: メンバに RAII 型を使えば特殊メンバ関数は不要
class Tracker {
    std::string name_;              // string が自身を管理
    std::vector<float> history_;    // vector が自身を管理
    std::unique_ptr<cv::Mat> buf_;  // unique_ptr が自身を管理
    // デストラクタ不要！コピー/ムーブも自動生成で OK
};

// Rule of Five が必要なケース: 生のリソース（FILE*, fd, デバイスハンドル）を直接持つとき
class CameraCapture {
    cv::VideoCapture cap_;  // 生のデバイスハンドル
public:
    ~CameraCapture();                              // 明示的解放が必要
    CameraCapture(const CameraCapture&) = delete; // デバイスは複製できない
    CameraCapture& operator=(const CameraCapture&) = delete;
    CameraCapture(CameraCapture&&) = default;     // ムーブ（所有権移転）は許可
    CameraCapture& operator=(CameraCapture&&) = default;
};

カメラキャプチャループの解剖

C++（main.cpp の骨格） 
int main() {
    // 1. CameraCapture を構築（デバイスはまだ開かない）
    media::CameraCapture cam;

    // 2. open() でデバイスを開く。カメラ ID は 0 が内蔵カメラ
    if (!cam.open(0)) {
        std::cerr << "カメラを開けませんでした\n";
        return 1; // ← open() を忘れると read_frame() は常に nullopt を返す
    }

    // 3. メインループ: 30fps 相当
    while (true) {
        auto frame = cam.read_frame(); // std::optional<cv::Mat>
        if (!frame) break;            // カメラが切断されたら終了

        cv::imshow("Camera", *frame);  // * でoptionalの中身を取り出す

        // 4. キーボードイベントを処理（GUI スレッドを動かすために必須）
        if (cv::waitKey(33) == 27) break; // 33ms 待機 = ~30fps / ESC で終了
    }
    // 5. CameraCapture のデストラクタが cap_.release() を自動で呼ぶ
}

Rule of Five と Rule of Zero

C++ 
class CameraCapture {
    // Rule of Five: デストラクタを自前で書いたら残り4つも定義する
    ~CameraCapture();                              // 自前で書く
    CameraCapture(const CameraCapture&) = delete; // コピー禁止を明示
    CameraCapture& operator=(const CameraCapture&) = delete;
    CameraCapture(CameraCapture&&) = default;     // ムーブはOK
    CameraCapture& operator=(CameraCapture&&) = default;
};

コピー禁止にする理由

カメラ（物理デバイスハンドル）を2つのオブジェクトが所有すると、両方のデストラクタが cap_.release() を呼び二重解放になります。= delete でコンパイルエラーにして型レベルで防ぎます。

例外安全○（try-finally）○（stack unwinding）
早期 return での安全△（書き忘れリスク）○（自動）
所有権の型表現△（慣習依存）○（= delete / unique_ptr）

HSV 色空間とは

OpenCV でよく使う HSV は BGR（Blue-Green-Red）より色抽出に向いた色空間です。

チャンネル意味OpenCV での値域物理的な意味
H（Hue / 色相）色の種類0〜179（本来の 0〜360° を半分にスケール）赤=0, 緑=60, 青=120, 赤=179（循環）
S（Saturation / 彩度）色の鮮やかさ0〜2550=灰色（無彩色）, 255=鮮やか
V（Value / 明度）明るさ0〜2550=黒, 255=最も明るい

BGR で「青っぽい色を抽出する」には B チャンネルが高く G・R が低い、という複雑な条件が必要です。HSV なら H を 100〜130 に絞るだけです。照明の変化（明るくなっても）は主に V に影響し、色 H はほぼ変わりません。

赤の検出が特殊な理由

H の値域が円環（0 と 179 が隣接）のため、赤は 0〜10 と 170〜179 の両端にまたがります。

C++（赤の二重 inRange） 
cv::Mat hsv;
cv::cvtColor(bgr, hsv, cv::COLOR_BGR2HSV);

cv::Mat mask1, mask2;
cv::inRange(hsv, cv::Scalar( 0, 80, 80),  cv::Scalar( 10, 255, 255), mask1); // 低端
cv::inRange(hsv, cv::Scalar(170, 80, 80), cv::Scalar(179, 255, 255), mask2); // 高端
cv::Mat red_mask = mask1 | mask2; // OR 合成

std::ranges パイプラインの詳細

std::ranges は TS の Array メソッドチェーン（filter, map）に対応します。最大の違いは「遅延評価」です。

TypeScript → C++ 対比 
// TypeScript: 各メソッドが配列を即時に生成（Eager）
const areas = contours
    .filter(c => cv.contourArea(c) > 100)  // 新配列を作る
    .map(c => cv.contourArea(c));            // また新配列を作る

// C++ std::ranges: 遅延評価（Lazy）— 中間配列を作らない
auto large_contours = contours
    | std::views::filter([](const cv::Mat& c) { return cv::contourArea(c) > 100; })
    | std::views::transform([](const cv::Mat& c) { return cv::contourArea(c); });
// ここではまだ計算されていない！
for (double area : large_contours) { // ← ここで初めて評価される
    std::cout << area << "\n";
}

遅延評価のメリット: 全要素を処理せず必要な分だけ計算できます。std::views::take(5) と組み合わせると最初の5要素だけを効率的に取り出せます。

std::span — コピーしない「借用ビュー」

C++ 
// std::span を使わない場合: vectorをコピーしてしまう
double max_area(std::vector<cv::Mat> contours) { // コピーが発生！
    ...
}

// span を使う: コピーなしで「借りる」(TS の ReadonlyArray<T> に近い)
double max_area(std::span<const cv::Mat> contours) { // ポインタ+サイズの2ワードのみ渡す
    auto it = std::ranges::max_element(contours,
        [](const cv::Mat& a, const cv::Mat& b) {
            return cv::contourArea(a) < cv::contourArea(b);
        });
    return it != contours.end() ? cv::contourArea(*it) : 0.0;
}

// 呼び出し側: vector, array, C配列など何でも渡せる
std::vector<cv::Mat> cs = find_contours(frame);
double m = max_area(cs); // 暗黙に span に変換される

cv::Mat の参照カウントと const& の関係

cv::Mat は内部で参照カウントを持ち、= 代入はシャローコピー（バッファ共有）です。const cv::Mat& で渡すと参照カウントも増えないため最も効率的です。独立コピーが必要な場合は .clone() を明示的に呼びます。

  • ムーブ: 所有権（内部リソース）を移転する。元は「空っぽ」になる。O(1) が多い
  • 参照（const T&）: 所有権も複製もしない。借りるだけ
  • cv::Mat は参照カウント付き。コピーしても実際はポインタのコピーのみ（浅いコピー）

Point a{ 1 };

Big-endian エンコードとは

同じ整数値でも、バイトをメモリに並べる順序が CPU アーキテクチャで異なります。ネットワークプロトコルは一般的に big-endian（最上位バイトが先）を使います。

// minimal_osc.hpp での変換

    return __builtin_bswap32(v); // バイト順を反転

requires 式の4種類

C++ 
template <typename T>
concept Detector = requires(T t, const cv::Mat& frame) {
    // 1. 単純要求: 「この式がコンパイルできること」
    t.process(frame);

    // 2. 型要求: 「この型が存在すること」
    typename T::output_type;

    // 3. 複合要求: 「この式がコンパイルでき、かつ戻り値型が一致すること」
    { t.process(frame) } -> std::same_as<std::optional<DetectorOutput>>;
    { t.name()         } -> std::convertible_to<std::string>;

    // 4. ネスト要求: 「この constraints が成立すること」
    requires std::copyable<T>;
};

制約の包摂（constraint subsumption）

    { t.process(f) } -> std::same_as<std::optional<DetectorOutput>>;
};

// より具体的な concept (Detector を包摂)
template <typename T>
concept GpuDetector = Detector<T> && requires(T t) {
    { t.supports_cuda() } -> std::same_as<bool>;
};

// オーバーロード: GpuDetector の方が具体的なので優先される
template <Detector T>    void run(T& d) { std::cout << "CPU\n"; }
template <GpuDetector T> void run(T& d) { std::cout << "GPU\n"; } // こちらが勝つ

Farneback オプティカルフロー アルゴリズムの概要

Farneback 法は「Dense」オプティカルフロー手法の一つです。2つのグレースケールフレーム間の全ピクセルの動きベクトルを計算します。

  1. 各ピクセル周辺を多項式（2次関数）で近似する
  2. 2フレーム間で多項式の係数を比較し、最も説明が一致する変位（dx, dy）を求める
  3. 結果は cv::Mat の2チャンネル（dx, dy）浮動小数点行列として返る

cv::calcOpticalFlowFarneback のパラメータ

C++ 
cv::Mat flow;
cv::calcOpticalFlowFarneback(
    *prev_gray_,  // 前フレーム（グレースケール）
    gray,         // 現在フレーム（グレースケール）
    flow,         // 出力: 2ch float mat (flow[y][x] = (dx, dy))
    0.5,          // pyrScale: ピラミッドのスケール比（0.5 = 各レベルで半分）
    3,            // levels: ピラミッドの段数。多いほど大きな動きを捉えられる
    15,           // winsize: 平均化ウィンドウサイズ。大きいほど滑らか・遅い
    3,            // iterations: 各ピラミッドレベルでの反復回数
    5,            // polyN: 多項式近似の近傍ピクセル数（5 or 7 が一般的）
    1.2,          // polySigma: ガウス標準偏差。polyN=5 なら 1.1, polyN=7 なら 1.5 推奨
    0             // flags: 0 でよい（OPTFLOW_USE_INITIAL_FLOW などのオプション）
);

// 動き量の平均を計算
cv::Scalar mean_flow = cv::mean(cv::abs(flow));
double magnitude = (mean_flow[0] + mean_flow[1]) / 2.0; // dx, dy の平均

False sharing — キャッシュライン競合

異なるスレッドが別々の変数を読み書きしているのに、それらが同じキャッシュライン（64バイト）に入っていると、CPUキャッシュの一貫性プロトコルが無駄にラインを無効化し合って性能が下がります。

C++ 
// NG: head_ と tail_ が同じキャッシュライン → false sharing
struct SpscQueue {
    std::atomic<size_t> head_{0};  // Producer が書く
    std::atomic<size_t> tail_{0};  // Consumer が書く
    // 両者が同じ 64B ライン → 互いに invalidate し合う
};

// OK: alignas(64) で別ラインに分離
struct SpscQueue {
    alignas(64) std::atomic<size_t> head_{0};
    alignas(64) std::atomic<size_t> tail_{0};
    // 別ラインに分かれて互いに干渉しない
};

happens-before とメモリモデル

「happens-before」とは「操作 A の効果が操作 B から必ず見える」という保証です。C++ のメモリモデルはこれを以下の手段で実現します。

同期手段happens-before 保証
mutex の lock/unlockunlock → lock の間に happens-before が成立
atomic + release/acquirerelease store → acquire load の間に happens-before が成立
std::atomic::store(seq_cst)全スレッドに対して全順序を保証（最も強い）
relaxed atomic原子性はあるが順序保証なし

SpscQueue の実装解剖

C++（SpscQueue の核心部分） 
template<typename T, size_t N>
class SpscQueue {
    std::array<T, N> buf_;
    alignas(64) std::atomic<size_t> head_{0}; // Producer が進める
    alignas(64) std::atomic<size_t> tail_{0}; // Consumer が進める

public:
    // Producer: データを書き込む
    bool try_push(T item) {
        size_t h = head_.load(std::memory_order_relaxed); // 自分の head を読む
        size_t next = (h + 1) % N;
        if (next == tail_.load(std::memory_order_acquire)) return false; // キュー満杯
        buf_[h] = std::move(item);
        head_.store(next, std::memory_order_release); // Consumer に「書き込み完了」を通知
        return true;
    }

    // Consumer: データを読み出す
    std::optional<T> try_pop() {
        size_t t = tail_.load(std::memory_order_relaxed);
        if (t == head_.load(std::memory_order_acquire)) return std::nullopt; // 空
        T item = std::move(buf_[t]);
        tail_.store((t + 1) % N, std::memory_order_release);
        return item;
    }
};

acquire ロードで「release ストア以前の書き込みが見える」が保証されます。これが happens-before の実現です。mutex を使わないため lock/unlock コストがなく、リアルタイム処理に適しています。

// and_then: 成功時に変換を適用（失敗はそのまま通過）
// std::expected<A, E> → (A → std::expected<B, E>) → std::expected<B, E>
load_config(path)
    .and_then([](Config cfg) -> std::expected<HsvRange, ConfigError> {
        if (cfg.hsv.lower[0] > cfg.hsv.upper[0]) {
            return std::unexpected(ConfigError{
                ConfigErrorKind::InvalidValue,
                "HSV lower > upper"
            });
        }
        return cfg.hsv;
    })
    .and_then([](HsvRange hsv) -> std::expected<Tracker, ConfigError> {
        return Tracker{hsv}; // 成功
    });

// or_else: 失敗時に回復処理（成功はそのまま通過）
// std::expected<A, E> → (E → std::expected<A, F>) → std::expected<A, F>
load_config(path)
    .or_else([](ConfigError err) -> std::expected<Config, ConfigError> {
        if (err.kind == ConfigErrorKind::FileNotFound) {
            return make_default_config(); // デフォルトで回復
        }
        return std::unexpected(err); // 回復不能はそのまま伝搬
    });

// transform: 成功値を変換（失敗はそのまま通過）
// std::expected<A, E> → (A → B) → std::expected<B, E>
load_config(path)
    .transform([](Config cfg) { return cfg.camera_id; }); // → expected<int, E>

即席 GUI（Immediate Mode GUI）とは

Dear ImGui は「即席モード（Immediate Mode）」GUI ライブラリです。従来の「保持モード（Retained Mode）」GUI（Qt、Win32 など）との根本的な違いがあります。

Retained Mode（Qt/Win32）Immediate Mode（ImGui）
状態の管理ウィジェットオブジェクトが内部状態を保持呼び出し元のデータが状態。ウィジェットは毎フレーム再生成
コード量多い（クラス・シグナル・スロット）少ない（関数を並べるだけ）
再利用性高い（UI コンポーネント化が容易）低い（テストが難しい）
向いている用途製品 UI、複雑なレイアウトデバッグパネル、パラメータ調整

std::mutex vs std::atomic の選び方

状況推奨理由
単純な数値（int, float, bool）の読み書きstd::atomic<T>mutex より軽量。多くの場合ロックフリー
複数フィールドをアトミックに更新std::mutex複数操作をまとめて保護
std::vector, std::string などのコンテナstd::mutexatomic が使えない（トリビアルでない型）
読み込みが多く書き込みが少ないstd::shared_mutex読み込みを並列化できる（C++17）
C++（atomic と mutex の使い分け） 
class UiState {
    mutable std::mutex mutex_;     // mutable: const メソッドでもロックできる
    std::atomic<float> fps_{0.0f}; // 単純な float → atomic で十分
    std::vector<std::string> logs_; // コンテナ → mutex が必要

public:
    // atomic は mutex 不要
    void set_fps(float fps) { fps_.store(fps, std::memory_order_relaxed); }
    float get_fps() const { return fps_.load(std::memory_order_relaxed); }

    // コンテナは mutex で保護
    void add_log(std::string msg) {
        std::lock_guard guard{mutex_}; // RAII ロック（スコープ終了で自動解放）
        logs_.push_back(std::move(msg));
    }
};

よくあるビルドエラー Q&A

error: use of undeclared identifier 'xxx'

ヘッダの #include 忘れ、または名前空間の指定忘れが多いです。使いたい型/関数のヘッダを確認してください。

undefined reference to 'xxx'

コンパイルは通るがリンクで失敗しています。target_link_libraries でライブラリのリンクが抜けているか、実装 (.cpp) がビルドに含まれていない可能性があります。

error: call to deleted function

= delete されたコピーコンストラクタ/代入演算子を呼ぼうとしています。std::move() でムーブするか、設計を見直してください。

error: no matching function for call to ...

オーバーロード解決に失敗しています。引数の型、const 修飾子、テンプレート制約（concept）を確認してください。
